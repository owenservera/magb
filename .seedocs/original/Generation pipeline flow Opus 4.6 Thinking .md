# magB Data Generation Pipeline
**Architectural Blueprint & Implementation Guide**

## 1. Overview & Philosophy
The magB Data Generation Pipeline is an automated, self-correcting engine designed to extract, structure, and verify technical knowledge from Large Language Models (LLMs). 

Rather than treating the LLM as a chatbot, the pipeline treats it as a **raw knowledge dataset**. The pipeline’s job is not to "teach" the AI about file formats or programming languages, but to execute **structured extraction with verification**—forcing the LLM to dump its latent knowledge into strictly typed, multi-resolution JSON schemas, and then testing that knowledge empirically.

### Key Architectural Decisions:
1. **Recursive Decomposition:** No hardcoded templates. The AI discovers the taxonomy of the target dynamically.
2. **Completeness Anchors:** Independent ground-truth lists (e.g., "all keywords", "all built-ins") are generated first and used as checklists to prevent silent AI omissions.
3. **Cross-Model Validation:** Content generated by one model (e.g., Claude 3.5 Sonnet) is validated by a different model (e.g., GPT-4o) to catch distinct hallucination patterns.
4. **Validation by Execution:** Generated implementation code is actually compiled/run in a sandbox. If it fails, the pipeline requests a correction.

## 2. The 12-Phase Generation Lifecycle

The pipeline operates in two major halves: generating the descriptive Reference Layer, followed by generating the executable Implementation Layer.

### Part A: The Reference Layer (Layer 1)
*Goal: Exhaustively document every feature, keyword, and syntax.*

* **Phase 1: Decompose Topic Tree**
  Recursively maps the target. Starts at the root ("Python") and asks for categories, then sub-categories, down to leaf nodes ("For Loop").
* **Phase 2: Enumerate Completeness Anchors**
  Generates exhaustive, verifiable ground-truth lists: every language keyword, every standard library module, every built-in function, and every operator.
* **Phase 3: Generate Reference Content**
  Massively parallel generation of content for every leaf node. Generates three resolutions (Micro, Standard, Exhaustive) and structured fields (Syntax, Parameters, Examples, Edge Cases).
* **Phase 4: Gap Analysis**
  Diffs the generated Topic Tree (Phase 1) against the Completeness Anchors (Phase 2). Detects missing nodes.
* **Phase 5: Fill Gaps**
  Automatically routes missing nodes back through Phase 3 to ensure 100% coverage.
* **Phase 6: Validate Accuracy**
  Statistically samples entries and sends them to a secondary LLM for fact-checking.

### Part B: The Implementation Layer (Layers 2 & 3)
*Goal: Extract exact structural atoms, mathematical algorithms, and application blueprints.*

* **Phase 7: Enumerate Capabilities**
  Translates the reference tree into actionable capabilities (e.g., "Draw a rectangle", "Apply Gaussian blur").
* **Phase 8: Extract Format Atoms**
  Extracts the irreducible structural units required for the capabilities (e.g., specific XML tags, binary hex headers, namespace URIs, coordinate system math).
* **Phase 9: Extract Algorithms**
  Generates full mathematical specifications, pseudocode, and multi-language implementations for any computational requirement (e.g., DEFLATE compression).
* **Phase 10: Generate Implementation Specs**
  Writes step-by-step instructions wiring the Atoms and Algorithms together to fulfill a Capability.
* **Phase 11: Assemble Blueprints**
  Combines multiple capabilities into full application architectures (e.g., "Complete architecture for a PPTX slide generator").
* **Phase 12: Validate Implementations**
  Extracts the generated code snippets and runs them. E.g., does the minimal PPTX code produce a file that opens without errors?

## 3. Core Engineering Mechanics

To survive the chaos of LLM API outputs at scale (generating ~5 million entries for 1,000 targets), the pipeline utilizes several heavy-duty engineering mechanics.

### A. The Response Parser (Recovery Engine)
LLMs frequently fail to return clean JSON. The pipeline uses a 5-strategy fallback parser to salvage API spend:
1. **Direct Parse:** Attempt standard `json.loads()`.
2. **Code Fence Extraction:** Regex matching for ` ```json ... ``` `.
3. **Boundary Extraction:** Finds the outermost `{ ... }` or `[ ... ]` brackets.
4. **Cleaned Parse:** Strips trailing commas, `//` comments, `/* */` blocks, and fixes single quotes.
5. **Truncation Repair:** If the LLM hit its max token limit mid-generation, the parser counts unclosed brackets/braces, strips broken trailing strings, and manually closes the JSON object to salvage the partial data.

### B. Multi-Tier Model Routing
To optimize the estimated $4,000–$8,000 total generation cost, the pipeline routes tasks by complexity:
* **Cheap Tier (e.g., gpt-4o-mini / claude-3-haiku):** Used for structural tasks like Phase 1 Decomposition and Phase 2 Enumeration.
* **Mid Tier (e.g., claude-3.5-sonnet):** Used for bulk content generation (Phase 3, Phase 8, Phase 9) due to its strong coding capabilities.
* **Expensive/Reasoning Tier (e.g., gpt-4o / o1):** Used selectively for Phase 6 Validation and resolving logical contradictions.

### C. The Deduplication Engine
When scaling to 1,000 targets, knowledge overlaps massively. The pipeline intercepts generation requests to enforce deduplication:
* If the pipeline is generating TIFF (Target #47) and needs the "DEFLATE Compression" algorithm, the Deduplication Engine checks the database.
* It recognizes DEFLATE was already generated for PNG (Target #3).
* Instead of regenerating the algorithm (saving tokens/time), it creates a graph edge: `target:tiff --uses_algorithm--> algorithm:compression:deflate`.

## 4. Orchestration, Safety & Resilience

Generating a single complex target (like Python or PPTX) requires ~2,500 API calls and takes 8-15 minutes. The orchestrator ensures this runs safely.

### Checkpoint & Resume
The pipeline maintains a `generation_checkpoints` SQLite table. Every completed API call and parsed result is logged. If the process crashes at step 1,842 of 2,500, restarting the CLI will instantly resume at step 1,843, preventing wasted API spend.

### Hard Budget Controls
The `BudgetExhausted` exception prevents runaway generation loops. The configuration enforces:
* `max_total_usd`: Hard limit for the whole CLI run.
* `max_per_target_usd`: Stops a single target from looping infinitely.
* `max_per_api_call_usd`: Sanity check to prevent accidental massive token context windows.

## 5. Wave-Based Rollout Strategy

To maximize the efficiency of the Deduplication Engine, targets are not generated in alphabetical order. They are generated in heavily curated "Waves" to seed foundational knowledge first.

* **Wave 1 (Foundation):** PNG, JSON, Python. 
  *(Establishes DEFLATE, text serialization, and core programming paradigms).*
* **Wave 2 (Related Formats):** JPEG, TIFF, JavaScript, YAML.
  *(Reuses 30-40% of Wave 1. E.g., JPEG reuses raster image concepts; JS reuses dynamic language paradigms).*
* **Wave 3 (Complex Assemblies):** PPTX, DOCX, XLSX, PDF.
  *(Heavily reuses ZIP/OPC concepts, XML structures, and DEFLATE).*
* **Wave 4 (Software Tools):** Photoshop, FFmpeg.
  *(Reuses almost all algorithms from Waves 1-3 to build massive application Blueprints).*
