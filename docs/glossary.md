# 📖 Glossary

Key terms used throughout the magB project.

---

| Term | Definition |
|---|---|
| **ACE** | AI Contribution Engine — system for community members to donate AI API credits to the project |
| **Algorithm** | A complete computational procedure stored in the database, including math, pseudocode, and working code implementations |
| **Anchor** | A completeness anchor — an independently-generated exhaustive list (e.g., all Python keywords) used to verify nothing is missing |
| **Atom** | The smallest structural unit of a file format (an XML element, a binary field, a JSON key) — the "periodic table" of file formats |
| **Blueprint** | An architecture plan that composes multiple capabilities and algorithms into a buildable application design |
| **Capability** | A user-facing feature of a technology, fully decomposed into implementable steps |
| **Coherence** | One of the 5 vital signs — measures whether knowledge entries connect properly (no orphans, no contradictions) |
| **Completeness Anchor** | See *Anchor* |
| **Concept** | A universal idea (e.g., "iteration", "compression") that spans multiple targets |
| **Content Resolution** | The level of detail for a knowledge entry: micro (~50 tokens), standard (~500), or exhaustive (~2000) |
| **Coverage** | One of the 5 vital signs — measures what fraction of a target's knowledge space is documented |
| **Decay** | The natural process by which knowledge becomes stale as the real world changes |
| **Delta Chain** | Version tracking strategy where each new version stores only what changed from the previous one |
| **Depth** | One of the 5 vital signs — measures whether all three knowledge layers are populated |
| **Entry** | A single piece of knowledge in the database, stored at three resolutions |
| **Exhaustive** | The most detailed content resolution (~2000 tokens) — for deep dives and implementation work |
| **Family** | A group of related targets (e.g., "C-family languages", "OPC-based formats") that share knowledge |
| **Freshness** | One of the 5 vital signs — measures how current knowledge is relative to real-world changes |
| **Generation Pipeline** | The multi-phase process that transforms a target name into a complete knowledge base |
| **Knowledge Base** | The complete structured output for a target — all entries, algorithms, atoms, blueprints, and relations |
| **Layer 1** | Capability Knowledge — "What can this technology do?" |
| **Layer 2** | Implementation Knowledge — "How does each feature actually work?" (atoms + algorithms) |
| **Layer 3** | Integration Knowledge — "How do I build complete systems?" (blueprints + composition rules) |
| **Micro** | The most condensed content resolution (~50 tokens) — for listings and summaries |
| **Observability** | The system that monitors all five vital signs of knowledge health |
| **Relation** | A typed, weighted edge in the knowledge graph connecting two entities |
| **Standard** | The mid-level content resolution (~500 tokens) — for most queries |
| **Target** | A specific technology being documented (e.g., "Python 3.12", "PPTX", "Photoshop") |
| **Vital Sign** | One of five dimensions of knowledge health: Coverage, Accuracy, Freshness, Depth, Coherence |
